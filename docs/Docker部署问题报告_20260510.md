## 三、问题详细记录

### 问题 #1：Docker 构建阶段缺少 unzip 工具

- **错误信息**：`unzip: not found`
- **根因**：Dockerfile 的 builder 阶段使用 `maven:3.9-eclipse-temurin-17` 基础镜像，该镜像未预装 `unzip`
- **修复方式**：在 builder 阶段添加 `RUN apt-get update && apt-get install -y unzip && rm -rf /var/lib/apt/lists/*`
- **涉及文件**：`java-backend/Dockerfile`

### 问题 #2：SLF4J 绑定冲突

- **错误信息**：`SLF4J: Class path contains multiple SLF4J providers` — log4j-to-slf4j AND slf4j-reload4j 同时存在
- **根因**：项目同时引入了 `logback-classic`（Spring Boot 默认）和 `slf4j-reload4j`（通过 redisson/rocketmq 传递依赖引入），两个 SLF4J 实现冲突
- **修复方式**：从 `spring-boot-starter-web` 中排除 `logback-classic`，让 `slf4j-reload4j` 作为唯一的 SLF4J 实现；同时创建 `log4j.properties` 配置文件
- **涉及文件**：`java-backend/pom.xml`、`java-backend/src/main/resources/log4j.properties`
- **经验教训**：Spring Boot 项目中 SLF4J 绑定只能有一个实现，传递依赖引入的日志框架需要通过 `<exclusion>` 排除或统一选择一个绑定

### 问题 #3：重复的 GlobalExceptionHandler Bean

- **错误信息**：Bean 定义冲突 — 两个 `GlobalExceptionHandler` 类
- **根因**：存在两个同名类：
  - `com.sjzm.common.GlobalExceptionHandler`
  - `com.sjzm.exception.GlobalExceptionHandler`
- **修复方式**：将 `RateLimitExceededException` 处理器合并到 `common.GlobalExceptionHandler`，删除 `exception.GlobalExceptionHandler`
- **涉及文件**：
  - `com/sjzm/common/GlobalExceptionHandler.java`（保留，增加处理器）
  - `com/sjzm/exception/GlobalExceptionHandler.java`（删除）

### 问题 #4：Bean 定义覆盖被禁止

- **错误信息**：`BeanDefinitionOverrideException: Invalid bean definition with name 'dataSourceProperties'`
- **根因**：`DataSourceConfig` 和 `DynamicDataSourceConfig` 都定义了 `dataSourceProperties()` Bean，Spring Boot 3.x 默认禁止 Bean 覆盖
- **修复方式**：在 `application.yml` 中添加 `spring.main.allow-bean-definition-overriding: true`
- **涉及文件**：`java-backend/src/main/resources/application.yml`
- **补充**：后续删除了 `DynamicDataSourceConfig` 中的重复 `dataSourceProperties()` Bean，从根源消除冲突

### 问题 #5：JdbcTemplate Bean 未创建

- **错误信息**：`No qualifying bean of type 'org.springframework.jdbc.core.JdbcTemplate'`
- **根因**：`DynamicDataSourceConfig` 创建了自定义 `dynamicDataSource`，但 Spring Boot 自动配置的 `JdbcTemplate` 依赖的是默认 `DataSource`，无法识别动态数据源
- **修复方式**：在 `DynamicDataSourceConfig` 中手动创建 `JdbcTemplate` Bean，注入 `dynamicDataSource`
- **涉及文件**：`com/sjzm/config/DynamicDataSourceConfig.java`

### 问题 #6：两个 @Primary DataSource 冲突

- **错误信息**：多个 @Primary DataSource Bean
- **根因**：`DataSourceConfig.hikariDataSource()` 和 `DynamicDataSourceConfig.dynamicDataSource()` 都标记了 `@Primary`
- **修复方式**：移除 `DataSourceConfig.hikariDataSource()` 的 `@Primary` 注解，保留 `dynamicDataSource` 作为主数据源
- **涉及文件**：`com/sjzm/config/DataSourceConfig.java`

### 问题 #7：Cache<String, Object> Bean 注入歧义

- **错误信息**：`NoUniqueBeanDefinitionException: No qualifying bean of type 'Cache<String, Object>'`
- **根因**：`CaffeineConfig` 定义了 5 个 `Cache<String, Object>` Bean（productCache、selectionCache、userCache、categoryCache、imageProxyCache），`MultiLevelCacheService` 无法确定注入哪一个
- **修复方式**：在 `MultiLevelCacheService` 的 `caffeineCache` 字段添加 `@Qualifier("productCache")`
- **涉及文件**：`com/sjzm/service/MultiLevelCacheService.java`

### 问题 #8：空壳 HikariDataSource 导致健康检查 503

- **错误信息**：`HikariCP-SJZM - dataSource or dataSourceClassName or jdbcUrl is required` + 健康检查返回 503
- **根因**：`DataSourceConfig.hikariDataSource()` 创建了 `HikariDataSource` 实例但只设置了 poolSize 和 poolName，没有设置 `jdbcUrl`/`username`/`password`，导致该数据源无法建立连接。Spring Actuator 的 DataSource 健康检查检测到此数据源异常
- **修复方式**：删除 `DataSourceConfig` 中的 `hikariDataSource()` Bean（该 Bean 本身是冗余的，`DynamicDataSourceConfig` 已正确创建了带完整配置的数据源）
- **涉及文件**：`com/sjzm/config/DataSourceConfig.java`
- **经验教训**：不要创建配置不完整的 DataSource Bean，Spring 会自动检测所有 DataSource 的健康状态

### 问题 #9：ProductController 重复端点映射

- **错误信息**：`IllegalStateException: Ambiguous mapping. Cannot map 'productController' method`
- **根因**：`ProductController` 中存在两个 `@GetMapping("/export")` 方法：
  - 第195行：`exportProducts()` 返回 `ResponseEntity<byte[]>`（Excel 文件下载）
  - 第313行：`exportProducts()` 返回 `Result<Map<String, Object>>`（JSON 数据导出）
  - 同一个 HTTP 方法 + 路径映射了两个处理方法
- **修复方式**：删除第313行的 JSON 版本导出方法（保留 Excel 文件下载版本，因为已有 `/export/by-skus` 和 `/export/template` 等完整的导出 API）
- **涉及文件**：`com/sjzm/controller/ProductController.java`

### 问题 #10：MySQL JDBC 字符集配置错误

- **错误信息**：`java.sql.SQLException: Unsupported character encoding 'utf-8mb4'` → `java.io.UnsupportedEncodingException: utf-8mb4`
- **根因**：JDBC URL 中 `characterEncoding=utf-8mb4` 是错误的。MySQL 的字符集名是 `utf8mb4`，但 JDBC `characterEncoding` 参数需要的是 Java 字符集名 `UTF-8`，不是 MySQL 字符集名。MySQL Connector/J 8.x 中 `UTF-8` 会自动映射到 MySQL 的 `utf8mb4`
- **修复方式**：将所有配置文件中的 `characterEncoding=utf-8mb4` 改为 `characterEncoding=UTF-8`（共4处）
- **涉及文件**：
  - `application.yml`（2处：主数据源 + 动态数据源 master）
  - `application-dev.yml`（1处）
  - `application-prod.yml`（1处）
- **经验教训**：JDBC 的 `characterEncoding` 参数使用 Java Charset 名（`UTF-8`），不是 MySQL charset 名（`utf8mb4`）

### 问题 #11：Docker 镜像名不匹配

- **错误信息**：docker-compose 启动时找不到镜像
- **根因**：`docker build` 构建的镜像名为 `sijue-java-backend:latest`，但 `docker-compose.yml` 中配置的镜像名为 `deploy-backend-java:latest`
- **修复方式**：每次构建后执行 `docker tag sijue-java-backend:latest deploy-backend-java:latest`
- **涉及文件**：`docker-compose.yml`
- **建议**：统一镜像名，将 Dockerfile 或 docker-compose.yml 中的名称对齐

### 问题 #12：RocketMQTemplate Bean 不可用

- **错误信息**：`No qualifying bean of type 'RocketMQTemplate'`
- **根因**：`RocketMQProducer` 使用 `@Autowired` 注入 `RocketMQTemplate`，但开发/测试环境中可能没有启动 RocketMQ Broker，导致该 Bean 不存在
- **修复方式**：
  1. 将 `@Autowired` 改为 `@Autowired(required = false)`
  2. 在所有使用 `rocketMQTemplate` 的方法中添加 null 检查，未初始化时跳过并记录 warn 日志
  3. 在 `application-prod.yml` 中添加 RocketMQ name-server 配置
- **涉及文件**：
  - `com/sjzm/mq/rocketmq/RocketMQProducer.java`
  - `application-prod.yml`

### 问题 #13：Docker Desktop PATH 问题

- **错误信息**：`无法将"docker"项识别为 cmdlet`
- **根因**：Windows 环境下 Docker CLI 未添加到 PowerShell 的 PATH 中
- **修复方式**：使用完整路径 `C:\Program Files\Docker\Docker\resources\bin\docker.exe` 执行命令

## 六、经验教训与建议

### 6.1 配置规范

1. **JDBC 字符集**：`characterEncoding` 必须使用 Java Charset 名（`UTF-8`），不是 MySQL charset 名（`utf8mb4`）
2. **Bean 命名**：多个同类型 Bean 必须使用 `@Qualifier` 指定注入目标，或使用 `@Primary` 明确主 Bean
3. **可选依赖**：中间件（RocketMQ、Redis 等）的注入应使用 `@Autowired(required = false)` + null 检查，确保单服务可独立启动

### 6.2 代码规范

1. **Controller 映射唯一性**：同一个 Controller 内禁止定义相同 HTTP 方法 + 路径的映射，IDE 应开启重复映射检查
2. **Bean 定义去重**：不要在多个 Config 类中重复定义相同 Bean，统一在一个地方定义
3. **DataSource 配置**：不要创建配置不完整的 DataSource Bean，Spring Actuator 会检测所有 DataSource 健康状态

### 6.3 部署规范

1. **镜像名统一**：Dockerfile 构建的镜像名应与 docker-compose.yml 中的 `image` 字段一致
2. **环境变量管理**：所有环境相关值走环境变量，禁止硬编码
3. **健康检查**：所有服务配置 healthcheck，确保依赖服务就绪后再启动上游服务

### 6.4 建议优化项

1. 将 Dockerfile 中的镜像名与 docker-compose.yml 对齐，避免每次手动 tag
2. 考虑统一日志框架为 logback（Spring Boot 默认），排除所有 slf4j-reload4j 传递依赖
3. 为 RocketMQ Broker 添加健康检查配置
4. 添加 Prometheus + Grafana 监控（当前 docker-compose.yml 已定义但未启用）
