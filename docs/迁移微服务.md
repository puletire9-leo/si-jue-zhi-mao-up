当前系统概览

  现状：单体的 Python FastAPI 后端，包含 ~28 个 API 模块、215 个接口，全部运行在一个 8080 端口的 uvicorn 进程上。Java    
  后端已经废弃清空（151 个文件被删，23651 行代码已移除）。

  值得微服务化的模块分析

  按优先级排序：

  P0 - 强烈推荐拆分

  1. 产品数据处理 (product_data) — Parquet 文件分析、销量趋势、对比分析
    - 原因：计算密集型（Polars 处理 97MB 数据），长时间查询容易阻塞其他 API
    - 独立后可以单独扩容、加缓存，不影响产品 CRUD 等轻量操作
  2. 图片管理 (images + image_proxy) — 上传、缩略图、相似搜索、代理
    - 原因：I/O 密集、占用带宽/内存，上传处理慢，有本地文件存储耦合
    - 独立后不会因为大图上传拖垮主进程

  P1 - 业务边界清晰，值得拆分

  3. 认证/用户 (auth + users) — JWT 登录、注册、角色权限
    - 原因：其他所有服务都依赖它，独立后可以统一认证网关
    - 但业务逻辑简单，拆分收益中等
  4. 选品 (selection + selection_recycle + scoring) — 选品工作流、评分
    - 原因：业务独立性强、逻辑复杂（评分算法）
    - 有自己的独立数据表，可以自洽运行
  5. 报告生成 (reports) — AI 报告生成
    - 原因：调用外部 AI 接口，耗时不可控，适合异步队列
    - 拆分后不会阻塞主流程

  P2 - 可拆可不拆

  6. 素材库/承运商/定稿 (material_library + carrier_library + final_drafts) — CRUD 为主的模块
    - 业务逻辑不复杂，单纯的 CRUD 收益不大
    - 除非未来需要独立部署给外部合作方使用
  7. 导入导出 (import + export + download_tasks) — 文件处理
    - 适合做成异步任务服务，但目前业务量不大

  不建议拆分

  8. 标签/分类/系统配置/公告/日志 (tags, categories, system_config, logs, announcement)
    - 过于简单、低频，拆分反而增加网络延迟

  建议路线

  渐进式，不要一锅端：

  单体 (现在) → 先拆 product_data + images → 再拆 auth → 其余看情况

  当前最值得动手的是 product_data（产品数据分析），因为：
  - 计算密集，16 个月数据、5000+ 产品，Polars 分析有明显资源争抢
  - 边界清晰（读 Parquet 文件 + Redis 缓存）
  - 独立部署对系统其他部分零影响
